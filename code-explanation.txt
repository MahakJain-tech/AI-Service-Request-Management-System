2. FRONTEND DESIGN & STYLING ENGINE
------------------------------------------------------------------------
* AESTHETICS:
  The styling has been optimized for visual excellence using vanilla CSS styling rules combined with Tailwind utility styles.
  - Harmony: Rich dark theme palette consisting of Sleek Black/Slate backgrounds, vibrant blue accents (#4f7fff) as the primary tone, and teal (#00d4aa) as secondary highlights.
  - Micro-animations: Custom hover effects, glowing action buttons, floating badges, and smooth fades ensure an alive, responsive user interface.
  - Glassmorphism: Utilizing background-blur backdrops (`backdrop-filter`) and thin borders, creating a premium modern dashboard experience.

* STYLING COMPILATION (Solving cdn.tailwindcss.com warnings):
  - In a standard production layout, dynamic compiler scripts like `cdn.tailwindcss.com` are discouraged since they cause browser warnings, slower client-side rendering, and frame origin failures under `file://` protocols.
  - We configured a Tailwind config file `tailwind.config.js` to extract and process all styling classes found across all HTML files.
  - Running `$env:PATH = 'C:\Program Files\nodejs;' + $env:PATH; npx tailwindcss -i input.css -o assets/css/tailwind.css --minify` generates a static minified CSS stylesheet.
  - All HTML files reference this static stylesheet local stylesheet `<link href="assets/css/tailwind.css" rel="stylesheet">`.
  - This completely eliminates CDN compiler warnings, speeds up load times, and fixes file security origin errors.

3. FRONTEND INTERACTIVE LOGIC (LAB 3 EVENT LISTENERS)
------------------------------------------------------------------------
To satisfy requirements for Lab evaluations, several specific interactive event handlers are bound directly:

- `onclick`: Triggers pre-submission audits and visual resets (e.g. `logSubmitClick()`, `approveAndSend()`).
- `onchange`: Reacts instantly to inputs. For example, changing the ticket priority dropdown dynamically updates the color class of the Priority indicator badge in `service-request.html`.
- `oninput`: Alters counters dynamically as characters are entered (e.g. counting the description length in real-time).
- `onkeyup`: Performs validation checks (e.g., verifying email syntax as the user types, or instantly searching/filtering rows in `ticket-management.html`'s console).
- `onmouseover` / `onmouseout`: Show and hide custom help tooltips dynamically when hovering cards.

4. BACKEND INFRASTRUCTURE (Node.js, Express, Mongoose)
------------------------------------------------------------------------
The backend is structured to handle API requests efficiently and securely.

* SERVER.JS:
  - Loads environment configurations using `dotenv`.
  - Mounts middleware: `cors()` to prevent cross-origin errors, and `express.json()` to parse JSON payloads.
  - Serves static directories (`frontend/` and root `/`) to resolve index and routing structures.
  - Seeds mock tickets, customers, and employees on startup if the MongoDB collection is empty.

* SCHEMAS (Models):
  - Ticket (`models/Ticket.js`): Schema containing fields for sequential ticketId, customer credentials, product, priority, category, description, attachments, status ('open', 'in-progress', 'closed'), assignedAgent, and SLA resolution timelines. Includes a pre-save mongoose hook to auto-increment code generation sequence.
  - Customer (`models/Customer.js`): Customer details (name, email, company, account types) used for database normalization rules.
  - Feedback (`models/Feedback.js`): NPS feedback details linked to the corresponding ticketId.

------------------------------------------------------------------------
5. SELF-HEALING ARCHITECTURE & API ENDPOINTS
------------------------------------------------------------------------
To ensure high resilience, we implemented a graceful database connection fallback and API route integration.

* DATABASE CONFIGURATION (db.js):
  - Uses `serverSelectionTimeoutMS: 5000` to prevent database connection hanging.
  - If the live MongoDB Atlas cluster is unreachable (due to dns, network, or firewalls), the catch block logs:
    "Entering local in-memory Mock DB mode. Express server remains operational."
    and continues execution instead of calling `process.exit(1)` and crashing the server.

* RESILIENT ROUTING (ticketRoutes.js):
  - A utility helper `isDbConnected()` evaluates the readyState status of Mongoose connection.
  - When connection is active, endpoints process queries directly in MongoDB Atlas.
  - When database is offline, endpoints dynamically switch to manipulate a persistent local array structure (`inMemoryTickets`, `inMemoryFeedback`). This means the server stays up and fully functional (CRUD operations return 200, update status, record ticket submissions, save feedback) even without a database process running!

------------------------------------------------------------------------
6. FRONTEND API INTEGRATION
------------------------------------------------------------------------
All client portals are connected directly to the backend Express server:

* SUBMIT REQUEST (`service-request.html` & `user-portal.html`):
  - Collects form variables and constructs a JSON payload.
  - Dispatches a POST request to `/api/tickets`.
  - On network connection loss, it falls back to saving requests inside the browser's `localStorage` key (`nexusCRM_offline_tickets`), ensuring offline support.

* CLIENT USER PORTAL (`user-portal.html`):
  - Fetches the active list of tickets from `/api/tickets`.
  - Filters rows matching the active logged-in customer's email.
  - Submits NPS ratings and notes via POST request to `/api/feedback`.

* AGENT DASHBOARD (`dashboard.html`):
  - Loads live ticket collections from `/api/tickets` to dynamically compute dashboard statistics (Open, In Progress, Closed tickets) and render Chart.js diagrams.
  - Putting ticket status changes are updated on the backend via PUT requests:
    - Moving Kanban cards: `kanbanDrop` issues PUT to `/api/tickets/:id` with the new status.
    - Resolving AI drafts: Approving a draft in AI studio calls PUT to update the ticket status to 'closed'.
    - Closing requests: Clicking "Close Request" immediately triggers a status update PUT on the backend.

7. HOW TO START & RUN THE APPLICATION
========================================================================
To launch the full stack environment:

1. Open a terminal inside the project directory:
   cd crm_project/backend
2. Run the Node Express server:
   node server.js
3. Open your browser and navigate to:
   http://localhost:5000
4. To test role-based authentication, click "Agent Login" or "User Portal" and use the credentials listed in the demo boxes:
   - Agent: agent@nexuscrm.com / agent123
   - Customer: user@example.com / user123
